### DB Schema Details
The Supabase database consists of four tables matching the product specifications:

#### Table: `pains`
Stores user-reported pain points.
```sql
create table public.pains (
  id bigint generated by default as identity primary key,
  text text not null,
  description text,
  status text not null default 'open', -- 'open', 'building', 'solved'
  votes integer not null default 1,
  tool_name text,
  tool_url text,
  tool_description text,
  tool_photo_url text,
  created_at timestamp with time zone default timezone('utc'::text, now()) not null
);
```

#### Table: `pain_votes`
Tracks anonymous user votes to prevent duplicate upvotes on the same device.
```sql
create table public.pain_votes (
  id bigint generated by default as identity primary key,
  pain_id bigint references public.pains(id) on delete cascade not null,
  session_id text not null,
  created_at timestamp with time zone default timezone('utc'::text, now()) not null
);
```

#### Table: `solutions`
Stores submitted solutions for pains waiting for admin moderation.
```sql
create table public.solutions (
  id bigint generated by default as identity primary key,
  pain_id bigint references public.pains(id) on delete cascade not null,
  tool_name text not null,
  tool_url text not null,
  tool_description text,
  contributor_name text,
  contributor_email text,
  photo_url text,
  status text not null default 'pending', -- 'pending', 'approved', 'rejected'
  created_at timestamp with time zone default timezone('utc'::text, now()) not null
);
```

#### Table: `tools`
Stores general community tool submissions waiting for admin moderation.
```sql
create table public.tools (
  id bigint generated by default as identity primary key,
  name text not null,
  url text not null,
  description text,
  category text not null default 'tool',
  contributor_name text,
  contributor_email text,
  photo_url text,
  status text not null default 'pending', -- 'pending', 'approved', 'rejected'
  created_at timestamp with time zone default timezone('utc'::text, now()) not null
);
```

## 2. Implementation Logs & Mobile Polish

### New Pages & Components
- **SubmitToolPage** (`/submit-tool`): Clean community submission form with responsive layout, dropzones, and category selects.
- **AdminPage** (`/admin`): Dashboard layout displaying pending lists, previews of uploaded screenshot thumbnails, and actions.
- **SearchOverlay** (`Ctrl+K`): Full-screen lookup covering pains and approved tools, suggesting keywords like "figma", "tabs", "browser", "QR".

### Mobile CSS Responsive Adjustments
To achieve standard mobile scaling (e.g. 375px/480px) and eliminate layout breaks and horizontal overflow:
1. **Fluid Containers**: Overrode `.content-container` at `max-width: 1024px` to use `width: 90%` ensuring automatic side margins.
2. **Flexible Navbar**: Restructured the header navigation on screens `< 768px` to center the logo and wrap links (`flex-wrap: wrap`) gracefully onto a second row, preventing text clipping.
3. **Hero Heading Wrap**: Removed `white-space: nowrap;` on `.hero__heading` in the mobile viewport to allow title strings to wrap.
4. **Stacked Form Actions**: Stacked buttons vertically (`flex-direction: column; width: 100%`) for modals (`.modal-actions`), submit tool page, and admin cards on devices `< 480px` to match screen borders.

## 3. Test Cases & Verification Plan

### Automated Test Cases (`src/tests/supabaseService.test.js`)
We installed `vitest` and created a mock suite to test all data layer procedures:
- `fetchPains()`:
  - Verifies successful mapping of pains, upvote states (per session ID), and pending solution badges.
  - Verifies empty list fallback in case database connection fails.
- `addPain()`:
  - Verifies insertion parameters and that a vote is auto-credited to the submitter.
- `toggleVote()`:
  - Verifies a vote is deleted and count decremented if already upvoted.
  - Verifies a vote is created and count incremented if not voted yet.
- `submitSolution()` / `submitTool()`:
  - Verifies photo attachments are dispatched to Supabase storage buckets and records are created with a `pending` status.
- `searchAll()`:
  - Verifies keyword match filter logic across both database tables.
- `approveSolution()` / `approveTool()`:
  - Verifies admin approval shifts solution statuses and correctly updates target pain rows.

Run the test suite locally using:
```bash
npm run test
```
